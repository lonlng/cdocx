**cdocx produces dramatically smaller files** by omitting the full default template
that python-docx ships with (stylesWithEffects, customXml, full numbering
definitions). `webSettings.xml` is now generated by default. Thumbnail is available
via opt-in API (`add_thumbnail()`). This is a design choice favoring minimal output
over Microsoft Office feature parity.

**Structural parity achieved**: cdocx equivalent documents now match python-docx
in all key XML node counts (paragraphs, runs, tables, text nodes, rPr, pPr).

## 1. Empty Document Baseline

### python-docx Empty (36,563 bytes)
Ships with a complete default template containing:
- `styles.xml` (349KB): Full style definitions including hundreds of built-in styles
- `stylesWithEffects.xml` (438KB): VML fallback styles for older Word versions
- `numbering.xml` (5.5KB): Default list numbering definitions
- `docProps/thumbnail.jpeg` (8.3KB): Preview thumbnail
- `customXml/`: Document properties wrapper
- `word/webSettings.xml`: Web view settings

### cdocx Empty (5,038 bytes)
Minimal valid DOCX with only essential parts:
- `styles.xml` (~6,500 bytes): 16 built-in styles (Normal, Heading1-9, Title, Subtitle, ListParagraph, Strong, Emphasis, Hyperlink)
- `settings.xml` (269 bytes): Basic document settings
- `fontTable.xml` (294 bytes): Minimal font references
- `theme/theme1.xml` (271 bytes): Minimal color theme
- `word/webSettings.xml` (153 bytes): Browser optimization settings
- No thumbnail by default (opt-in via `add_thumbnail()`), no customXml, no stylesWithEffects
- 16 ZIP entries

**Verdict**: cdocx's minimal approach is superior for programmatic generation where
template files are created and destroyed. The python-docx bloat adds ~32KB of
unneeded data to every file.

### Numbering.xml Support Implementation
**Files**: `src/document.cpp`, `src/paragraph.cpp`, `src/section.cpp`
**Changes**:
1. `Document::save_numbering()` generates `word/numbering.xml` from the `NumberingManager` and adds the relationship to `document.xml.rels`.
2. `Document::create_empty()` now calls `load_all_relationships()` after `create_empty_document()` so the in-memory `relationships_` map is populated with initial rId1-rId4. This prevents subsequent `add_relationship()` calls from generating colliding IDs.
3. `Paragraph` list methods (`set_numbering`, `remove_numbering`, `set_list_level`) now update `list_format_` (DOM) as the primary source of truth, ensuring round-trip fidelity.
4. Fixed relationship ID mismatch in `Section::add_header()`, `add_footer()`, and `link_to_previous()` unlink paths. These functions now capture and use the return value of `add_relationship()` as the relationship ID instead of generating an independent ID.

**Results**:
- Bulleted, numbered, and outline lists survive save/reopen cycles.
- `word/numbering.xml` is correctly packaged in the DOCX ZIP.
- Header/footer relationship IDs remain consistent between `sectPr` and `document.xml.rels`.
- All 18 cdocx tests pass without regression.
